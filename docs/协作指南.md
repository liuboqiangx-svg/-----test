# 纸片人男友 - 开发协作指南

> 本指南是 Claude Code 的开发协作规范，每次开发开始前必须阅读。

---

## 一、核心原则

### ⚠️ 开发前必读

每次开发开始前，必须按顺序完成：

```
1. 读取 docs/开发计划.md → 了解当前进度和下一步
2. 读取 docs/协作指南.md → 确认协作规则
3. 完成开发前准备检查清单（见第三章）
4. 确认用户理解后再开始
```

### 五大核心规则

| 规则 | 说明 |
|------|------|
| **双重存档** | 开发计划同时保存在 `docs/开发计划.md` 和对话 Todo 中 |
| **建议模式** | 遇到问题时，提供 2-3 个选项 + 分析利弊 |
| **原理优先** | 每一步解释"为什么这样做"，带形象比喻 |
| **细粒度检查** | 每一步代码前检查前置依赖是否完成 |
| **自动恢复** | 对话中断后，读取 `docs/开发计划.md` 恢复上下文 |

## 二、沟通与反馈

### 2.1 遇到问题时

**方式**：给选项 + 分析利弊，让用户选择

**格式**：
```
问题描述：A 方案可能导致 XX 问题

选项：
A. 方案A（优点：XX，缺点：XX）
B. 方案B（优点：XX，缺点：XX）
C. 方案C（优点：XX，缺点：XX）

💡 建议：推荐 B，因为 XX
```

### 2.2 代码解释深度

**级别**：完整解释 = 做什么 + 怎么实现 + 为什么不那样做 + 形象比喻

**示例**：
```typescript
// 问题：为什么要用 useState？
useState 是 React 的状态管理钩子。
- 做什么：保存会变化的变量，当它变化时自动更新界面
- 怎么实现：返回一个 [值, 更新函数] 的数组
- 为什么不用普通变量：普通变量变化不会触发重新渲染
- 比喻：就像给变量装了一个"变形开关"，一变界面就跟着变
```

### 2.3 遇到不会的概念时

**方式**：类比 + 图示 + 例子三合一

**格式**：
```
概念：什么是 API？

1. 比喻：API 就像餐厅的点餐系统
   - 你（前端）不用进厨房
   - 只需告诉服务员（API）你要什么
   - 厨房准备好了会送出来

2. 图示：
   [用户] → [你的请求] → [API] → [处理] → [返回结果] → [你]

3. 代码例子：
   fetch('/api/user')  // 调用 API
     .then(res => res.json())  // 获取返回的数据
```

### 2.4 风险预警

**方式**：完整呈现风险 + 方案 + 建议

**格式**：
```
⚠️ 潜在风险：XX

方案A：XX
  - 优点：XX
  - 缺点：XX

方案B：XX
  - 优点：XX
  - 缺点：XX

💡 建议：推荐方案B，因为 XX

请选择：___
```

### 2.5 进度汇报

**方式**：直播式随时同步

**示例**：
```
📝 正在创建 image generator 模块...

✅ 步骤1：定义类型接口
✅ 步骤2：封装火山引擎 API 调用
🔄 步骤3：实现 Prompt 组装器（进行中...）
⏳ 步骤4：添加错误处理
⏳ 步骤5：写测试用例
```

### 2.6 沟通风格

**风格**：稍微详细，有条理

- 使用 ✅/🔄/⏳/⚠️ 等符号表示状态
- 每个步骤用序号标注
- 重要结论用 **粗体** 强调
- 适当使用 emoji，但不过度

## 三、开发前准备

### 3.1 前置检查清单

每开始一个新任务前，必须逐一确认：

```
□ 功能目标确认（用户想要什么效果？）
□ 输入/输出确认（数据怎么进、怎么出？）
□ 涉及模块确认（在哪个文件/目录改？）
□ 异常处理确认（出错了怎么办？）
□ 边界情况确认（极限情况考虑了吗？）
□ API/接口确认（涉及外部调用时）
□ 类型定义确认（涉及数据结构时）
□ 用户理解确认（"我理解的是...对吗？"）
```

### 3.2 需求确认方式

根据问题复杂度，灵活选择：

#### 小问题/小功能 → 使用结构化模板

**五项模板**：
```
【功能目标】
要做的事：___

【输入/输出】
输入：___
输出：___

【涉及模块】
文件/目录：___

【异常处理】
错误情况：___ → 处理方式：___

【边界情况】
极端情况1：___ → 处理方式：___
极端情况2：___ → 处理方式：___
```

#### 复杂问题 → 使用苏格拉底提问法

**提问原则**：
- 从"目的"开始追问
- 每个问题带序号
- 追问到出现具体实现方向就停止
- 问完后总结 + "还有什么要补充的吗？"

**示例**：
```
📝 需求梳理 - 图像生成功能

问题1：你希望这个功能最终实现什么效果？
回答：让角色能主动发送照片

问题2：为什么需要这个效果？
回答：因为用户想要看到角色的样子

问题3：用户想看到什么样的"样子"？
回答：能看到角色的各种表情和场景

→ 已明确实现方向：表情变化 + 场景变化

💡 我的理解：
要做的是一个图像生成功能，能根据角色情绪和场景
生成对应的图片。核心是表情映射 + 场景组合。

还有其他要补充的吗？
```

### 3.3 用户理解确认

每完成一个阶段，用以下方式确认：

```
✅ 阶段完成：图像生成模块

📋 已实现：
- 基础形象生成
- 表情变化生成
- 场景变化生成

❓ 请确认：
- 功能是否符合预期？
- 还有需要调整的地方吗？
```

## 四、开发流程

### 4.1 里程碑确认机制

#### 小里程碑（自我检查）

每个功能模块完成后自问：
- 这段代码能独立运行吗？
- 符合预期吗？
- 有没有明显的 bug？

#### 大里程碑（用户验收）

每个大阶段完成后停下来：
1. 总结已完成的内容
2. 列出下一步计划
3. 请用户确认后再继续

### 4.2 断点续接规则

对话中断后恢复时：

```
1. 读取 docs/开发计划.md
2. 告诉用户：
   "📋 上次进度：XX
    📍 当前位置：XX
    🎯 下一步：XX"
3. 等待用户确认后继续
```

### 4.3 代码改动记录

每次代码改动必须记录：

**格式**：
```markdown
## [日期] XX功能改动

### 改了什么
- 文件：A.tsx
- 函数：updateImage()
- 改动：添加了新的参数支持

### 为什么改
- 原因：需要支持多角色

### 有什么影响
- 影响：旧代码兼容（需要额外处理）
- 风险：中等（已测试）

### 验证方式
- 测试用例：覆盖了A、B、C场景
```
